Reverse Engineering a Marketplace API: 5 Versions, 2 Years, 2100 Lines

How I went from copying cURL commands from browser DevTools to building a production-grade, multi-account extraction system with cookie-based auth, anti-detection, and parallel processing.

Reverse Engineering Python APIs Anti-Detection

Why Reverse Engineer?

Wallapop is Spain's largest second-hand marketplace. It doesn't have a public API. No OAuth docs, no developer portal, no rate limit documentation. But my business depended on it — I needed to extract orders, track shipments, analyze conversations, and manage listings across multiple accounts.

So I opened Chrome DevTools and started watching network requests.

The Evolution: V1 → V3

V1: The Naive Approach

The first version was embarrassingly simple: hardcode a bearer token, make requests to the endpoints I found in DevTools, parse the JSON. It worked for about a week before the token expired.

The token management was manual — I'd open Wallapop in the browser, copy the bearer from the Network tab, paste it into a JSON config file. For one account, this was annoying. For the 12+ accounts we eventually operated, it was unsustainable.

V2: Bearer Management + Multi-Account

V2 introduced two key innovations:

  • Bearers stored in Google Sheets — Instead of local JSON files, tokens went into a shared Sheet that multiple scripts could read. Each row: account name, bearer token, user hash, last validated timestamp
  • Validation before use — Before making API calls, the system would hit a lightweight endpoint (/bff/messaging/inbox) to verify the token was still valid. If not, it'd flag the account for manual token refresh

I built a Tkinter UI (yes, desktop Python GUI) that my employees could use to paste new bearers when they expired. Not elegant, but functional.

V2.5: Cookie-Based Auth

The breakthrough came when I realized I didn't need bearers at all. Wallapop's web app uses cookie-based sessions. If I could capture and refresh the session cookies, I could maintain auth indefinitely without manual intervention.

The CookieManager (~990 lines) handles:

  • Cookie storage in Google Sheets (later PostgreSQL)
  • Automatic accessToken refresh using the session cookie
  • JWT decoding to check expiration times
  • Validation against the API before each extraction run

This eliminated 90% of the manual token management. Accounts would stay authenticated for weeks instead of hours.

V2.6: Parallel Processing

With 12+ accounts, sequential extraction was too slow. V2.6 introduced ThreadPoolExecutor for parallel multi-account extraction. Each account gets its own thread, its own session, and its own rate limiting.

V3: Bundles and Complex Order Logic

Wallapop introduced "bundles" — orders that contain multiple items from the same seller. This broke my entire order processing pipeline because the API response structure changed: one delivery could contain multiple items, each with its own LPN code that needed to be extracted from the product description using regex.

V3 (~748 lines for the manager alone) handles bundle detection, LPN extraction via regex patterns, warehouse location enrichment from comments, and cross-referencing with the preparation Excel from Google Drive.

Anti-Detection Engineering

When you're making thousands of API calls per day across multiple accounts, you need to not look like a bot. Here's what I built:

  • User-Agent rotation — Pool of 50+ real browser UAs, rotated per-request, stored in PostgreSQL
  • Request timing — Random delays between requests (not fixed intervals, which are a detection signal)
  • Header fingerprinting — Exact replication of browser request headers, including order, casing, and optional headers
  • Session consistency — Same UA and cookie set for the duration of a session, switching only between runs
  • Image perturbation — For uploaded product images, OpenCV transformations (noise, micro-rotation, stretch) to avoid perceptual hashing detection of "duplicate" listings across accounts

The Architecture Today

Extractor Architecture
BaseExtractor (ABC)
├── Anti-detection: UA rotation, header fingerprint, timing
├── Session management: cookie refresh, JWT validation
├── Error handling: retries, backoff, account-level circuit breaker
│
├── OrdersExtractor (~2100 lines)
│   ├── 20+ order state machine transitions
│   ├── Bundle detection and LPN extraction
│   ├── Sale creation with price/fee/net calculation
│   └── Withdrawal tracking for returns
│
├── ChatsExtractor
│   ├── Conversation pagination
│   ├── Message extraction with threading
│   └── Async Sheets writer with batching
│
└── ListingsExtractor
    ├── Stats accumulation (views, favorites, conversations)
    ├── Anti-oscillation for product_id changes
    └── Price and status change tracking

The orders extractor alone is 2,100 lines because marketplace order logic is genuinely complex. An order can be: pending acceptance, accepted, shipped, delivered, completed, cancelled, returned, disputed, or stuck in any intermediate state. Each transition needs to update the right tables and trigger the right downstream actions.

Lessons Learned

  1. Version your architecture aggressively. I kept all 5 versions because going back to V2 when V2.5 had a bug was a real scenario. The naming convention (V1, V2, V2.5, V2.6, V3) tells a story of incremental evolution
  2. Anti-detection is an arms race. What works today won't work in 6 months. Build your system to be modular enough that you can swap detection-avoidance strategies without rewriting the extraction logic
  3. Invest in auth management early. I spent weeks on manual bearer token management that could have been avoided if I'd built the cookie-based system from the start
